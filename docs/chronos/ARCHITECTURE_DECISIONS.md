# ChronOS — Architectural Decisions Document (ADR)

This document records the foundational architectural decisions for ChronOS, outlining technical rationales, constraints, and implementations.

---

## 1. Authentication & Tenant Isolation

### Decision
Use **Supabase Auth** for JWT verification and PostgreSQL **Row Level Security (RLS)** for multi-tenant data isolation.

### Implementation Details
- Every table containing user data will have a `user_id` column.
- All RLS policies will explicitly define both `USING` and `WITH CHECK` policies to restrict read, write, update, and delete access to `auth.uid() = user_id`.
- A database trigger on `auth.users` will automatically create a matching profile in `public.user_profiles` immediately upon registration.

### Auto-Profile Creation Trigger (PostgreSQL)
```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO public.user_profiles (id, display_name, timezone, working_hours_json, focus_preferences_json, autonomy_level)
    VALUES (
        new.id,
        COALESCE(new.raw_user_meta_data->>'display_name', 'ChronOS User'),
        'UTC',
        '{"start": "09:00", "end": "17:00", "days": [1,2,3,4,5]}'::jsonb,
        '{"deep_work_duration": 90, "shallow_work_duration": 30, "default_buffer_percent": 15}'::jsonb,
        'ask'::autonomy_level_type
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE OR REPLACE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

## 2. Google OAuth Separation

### Decision
Decouple Google OAuth token storage from the core `user_profiles` table. Create a dedicated `google_connections` table to store Google account links and credentials, preventing profile read operations from loading credentials unnecessarily.

### Schema Definition
```sql
CREATE TABLE google_connections (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES user_profiles(id) ON DELETE CASCADE UNIQUE NOT NULL,
    google_email TEXT NOT NULL,
    access_token TEXT NOT NULL,          -- Encrypted in application layer
    refresh_token TEXT NOT NULL,         -- Encrypted in application layer
    token_uri TEXT NOT NULL,
    client_id TEXT NOT NULL,
    scopes TEXT[] NOT NULL,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL
);

ALTER TABLE google_connections ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can CRUD their own Google connection"
    ON google_connections FOR ALL 
    USING (auth.uid() = user_id)
    WITH CHECK (auth.uid() = user_id);
```

## 3. Dedicated Table for Decision Dock Approvals

### Decision
Introduce a dedicated `agent_proposed_actions` table to store actions generated by the Replanner, Rescue, or Negotiator agents. These actions remain in a "pending" status until the user approves or rejects them through the Decision Dock.

### Schema Definition
```sql
CREATE TYPE proposed_action_type AS ENUM ('calendar_create', 'calendar_move', 'calendar_delete', 'commitment_reschedule', 'commitment_rescue', 'negotiation_send');
CREATE TYPE proposed_action_status AS ENUM ('pending', 'approved', 'rejected', 'expired');

CREATE TABLE agent_proposed_actions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES user_profiles(id) ON DELETE CASCADE NOT NULL,
    agent_run_id UUID REFERENCES agent_runs(id) ON DELETE CASCADE NOT NULL,
    action_type proposed_action_type NOT NULL,
    status proposed_action_status DEFAULT 'pending'::proposed_action_status NOT NULL,
    payload_json JSONB NOT NULL, -- Target action variables (e.g. event_id, start_time, draft_text)
    explanation TEXT NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL
);

ALTER TABLE agent_proposed_actions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can CRUD their own proposed actions"
    ON agent_proposed_actions FOR ALL 
    USING (auth.uid() = user_id)
    WITH CHECK (auth.uid() = user_id);
```

## 4. Compound Unique Constraint for External Events

### Decision
Replace the global `google_event_id UNIQUE` constraint with a compound `UNIQUE (user_id, google_event_id)` constraint. This prevents event ID collisions across different users and prevents user data leakage.

## 5. Multi-Mode Agent Architecture

### Decision
Do not implement a single, monolithic LangGraph. Instead, split agent operations into six lightweight, dedicated graphs. This keeps token usage low, simplifies testing, and reduces LLM response latency.

### Graph Modes
1. **`intake_plan`**: Takes a messy text dump, extracts commitments, compiles next actions, computes initial risk, and creates default Time Spine structures.
2. **`drift_replan`**: Triggered when a drift event is logged. Detects schedule conflicts and suggests adjustments (shifts, compression) to the Decision Dock.
3. **`rescue`**: Triggered when a commitment enters Critical risk. Generates a Minimum Viable Completion Path (MVCP) and sequence checklist.
4. **`negotiate`**: Generates customized communication templates for rescheduling or extensions based on the commitment context.
5. **`reflect_update_memory`**: Runs during review windows. Compares estimates with actual duration records to update personalized coefficients in memory.
6. **`calendar_refresh`**: Runs in the background to sync event data and identify available time slots.

## 6. AI Output Validation & Self-Repair Loop

### Decision
All Gemini API outputs must be validated against Pydantic schemas. If validation fails, the system executes exactly one repair attempt.

```text
  [Gemini Model Output] ───> [Validate with Pydantic]
                                    │
                                    ├───> Success: Save to Database
                                    │
                                    └───> Fail: [Repair Loop]
                                                │
                                                ├───> Resend validation error to Gemini
                                                │
                                                └───> [Revalidate Output]
                                                            │
                                                            ├───> Success: Save
                                                            │
                                                            └───> Fail: Return Safe Default & Log Trace
```

### Self-Repair Implementation Flow
1. Receive model output.
2. Run `PydanticValidationError` check.
3. On failure, construct a prompt containing the error logs and ask the model to fix the output structure.
4. If validation fails a second time, reject the response, log a critical error in `agent_trace_events`, and return a safe, structured fallback object (e.g., an empty list of commitments or a stable mock state) to prevent system crashes.

## 7. Real-Time Agent Progress Monitoring

### Decision
Use **Server-Sent Events (SSE)** via FastAPI's `EventSourceResponse` to stream agent progress directly to the frontend. If SSE is unavailable, fall back to polling the `/api/v1/agent/runs/{id}/trace` endpoint every 1000ms.

### Implementation Details
- Every step in a LangGraph execution calls a logger service to insert records into the `agent_trace_events` table.
- The SSE endpoint streams these trace database records to the client in real time, showing the user exactly what the agent is doing.
- No dummy logs or client-side fake loading bars will be used. The frontend Agent Console will display real, server-side trace records.
